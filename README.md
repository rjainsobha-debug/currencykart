# CurrencyKart

Premium forex and travel-services MVP for Delhi NCR customers. The brand name is configurable in `config/brand.ts`.

## Launch branding

`CurrencyKart` is a fallback name only. Configure launch identity through:

- `NEXT_PUBLIC_BRAND_NAME`
- `NEXT_PUBLIC_LEGAL_NAME`
- `NEXT_PUBLIC_SITE_URL`
- `NEXT_PUBLIC_SUPPORT_PHONE`
- `NEXT_PUBLIC_SUPPORT_EMAIL`
- `NEXT_PUBLIC_WHATSAPP_URL`
- `NEXT_PUBLIC_REGISTERED_ADDRESS`

Before publishing:

- Replace all fallback legal, address and contact values.
- Publish verified authorisation and fulfilment-partner details only after documentary review.
- Review Terms, Privacy, Refund/Cancellation and KYC policies with Indian legal counsel.
- Confirm the final cancellation charges, refund timelines, data-retention schedule and grievance process.
- Replace the default social-preview image if final brand artwork is available.

## SEO and launch content

- Every public service and policy page has unique metadata and a canonical URL.
- Open Graph and Twitter preview metadata use the configured site URL and brand.
- `sitemap.xml` and `robots.txt` are generated by Next.js metadata routes.
- The homepage includes a LocalBusiness schema placeholder.
- The FAQ page includes FAQPage structured data.
- Local service-area content covers Delhi NCR, Delhi, Gurgaon, Noida, Faridabad and Ghaziabad.
- Auth, dashboard, admin and API routes are excluded from search crawling.

Launch content pages:

- `/terms`
- `/privacy`
- `/refund-cancellation`
- `/kyc-document-policy`
- `/compliance`

## What is included

- Next.js App Router, TypeScript and Tailwind CSS
- Premium public pages for buy forex, sell forex, forex card, insurance, student forex, corporate forex, about, contact and FAQ
- Customer dashboard and role-based admin dashboard surface
- NextAuth/Auth.js setup with email/password credentials and Google OAuth configuration
- Mobile OTP, email, payment, WhatsApp and storage provider placeholders
- Prisma PostgreSQL schema with KYC, documents, orders, payments, leads, support tickets, rates and audit logs
- Seed script for 25 requested currencies
- Zod validation for key API routes
- Secure upload checks for file type and size
- Guarded order lifecycle for buy, sell, forex card and card reload requests
- Dynamic document checklists by service and travel purpose
- Rate locks with expiry timestamps and customer countdown UI
- Admin order, KYC, payment, fulfilment and invoice actions with audit logs
- Customer order events/timeline and invoice download placeholder
- Filterable admin order queue and WhatsApp lead capture endpoint

## Provider adapters

Production adapter skeletons are under `lib/providers`:

- `sms/msg91.ts`
- `sms/twilio.ts`
- `email/ses.ts`
- `email/sendgrid.ts`
- `whatsapp/cloud-api.ts`
- `payment/razorpay.ts`
- `storage/s3.ts`

The provider factory in `lib/providers/index.ts` selects adapters using environment variables. `placeholder` remains the default for local development.

Provider selections:

```text
OTP_PROVIDER=placeholder|msg91|twilio
EMAIL_PROVIDER=placeholder|ses|sendgrid
WHATSAPP_PROVIDER=placeholder|cloud-api
PAYMENT_PROVIDER=placeholder|razorpay
STORAGE_DRIVER=placeholder|local|s3
```

Provider-specific variables are documented in `.env.example` and validated conditionally by `npm run env:validate`.

### Razorpay webhook

Configure the live webhook endpoint:

```text
https://currencykart.in/api/payments/razorpay/webhook
```

Set `RAZORPAY_WEBHOOK_SECRET` to the matching dashboard secret. The route validates the raw payload signature, records webhook idempotency state and audits successful payment updates.

### OTP and administrator 2FA

- OTP challenges expire after five minutes.
- Resends have a 60-second cooldown.
- Challenges allow five verification attempts.
- Verified phone-login challenges are single-use.
- Requests, failures, verification and session consumption are audited.
- Set `ADMIN_2FA_REQUIRED=true` to require staff OTP verification before protected admin API actions.

## Order operations

The lifecycle supports:

`DRAFT -> SUBMITTED -> KYC_PENDING -> KYC_APPROVED -> RATE_LOCKED -> PAYMENT_VERIFIED -> PROCESSING -> OUT_FOR_DELIVERY -> COMPLETED`

Rejection and cancellation branches are available where operationally appropriate. Admin actions are exposed through:

```text
POST /api/admin/orders/{orderId}/actions
```

Supported actions include order approval/rejection, KYC approval/rejection, rate locking, payment verification/rejection, processing, delivery, completion, cancellation and invoice generation.

Every admin action creates:

- An `OrderEvent` for the customer/admin timeline
- An `AuditLog` with old/new values and optional IP address
- A relevant email notification placeholder where applicable

Useful workflow endpoints:

- `GET /api/admin/orders` with status, currency, customer, payment and date filters
- `GET /api/orders/{orderId}` for order, documents, payment, timeline and rate-lock state
- `GET /api/document-checklist?orderType=BUY_FOREX&purpose=Student%20education`
- `POST /api/leads/whatsapp`

### Database-free UI preview builds

The public website, metadata, sitemap and robots routes do not query PostgreSQL
during `next build`. Prisma is initialized lazily, and authenticated dashboards
plus database-backed API routes are rendered only when requested at runtime.

For an initial UI-only Vercel preview, the build can use a syntactically valid
placeholder without connecting to it:

```text
DATABASE_URL=postgres://user:pass@db.example.com:5432/app
```

Use this only for UI preview deployments. Login, dashboards, orders, KYC,
payments and document workflows require a reachable Neon `DATABASE_URL` at
runtime. Configure the real Neon value before testing those functions.

Cloudflare DNS steps:

1. Open the `currencykart.in` zone in Cloudflare.
2. Add a DNS record:

   ```text
   Type: CNAME
   Name: staging
   Target: cname.vercel-dns.com
   Proxy status: DNS only
   ```

3. Keep Cloudflare proxy set to **DNS only** first.
4. Wait until Vercel verifies the domain.
5. Test:

   ```bash
   curl -I https://staging.currencykart.in
   curl -I https://staging.currencykart.in/robots.txt
   curl -I https://staging.currencykart.in/sitemap.xml
   ```

6. After HTTPS and routing are stable, decide whether to enable the Cloudflare proxy based on Vercel domain guidance and the desired WAF/CDN setup.

## Security checklist

- Keep document storage private and use signed upload/download URLs.
- Validate MIME type and file size server-side.
- Hash passwords with bcrypt or a stronger approved equivalent.
- Enforce role checks in every admin API route.
- Add Redis-backed rate limiting for auth, OTP, order and upload endpoints.
- Log sensitive admin actions in `AuditLog`.
- Add admin 2FA before production.
- Use CSRF/session protections from NextAuth/Auth.js.
- Never expose PAN, passport, visa or ticket files publicly.
- Review dependency versions before production release.

Dependency review:

```bash
npm audit
npm audit --omit=dev
```

Do not run `npm audit fix --force` without reviewing breaking-version changes. Record accepted findings and compensating controls in the release notes.

## Production readiness checklist

- Set a PostgreSQL connection URL with TLS requirements appropriate to the provider.
- Generate and commit the initial Prisma migration from a disposable development database.
- Run `npm run env:validate` in CI and before application startup.
- Replace the in-memory rate limiter with a Redis-backed implementation and set `RATE_LIMIT_STORE=redis`.
- Replace placeholder implementations in `lib/providers/placeholder.ts`.
- Configure Google OAuth callback URLs for the production domain.
- Keep document and invoice buckets private; store object keys only.
- Use short-lived signed upload and download URLs.
- Configure structured log collection and alerting for `error` events.
- Protect admin accounts with provider-backed 2FA.
- Rotate `NEXTAUTH_SECRET`, admin bootstrap credentials and provider secrets.
- Disable or remove demo users before launch.
- Verify database backups, point-in-time recovery and migration rollback procedures.
- Add payment webhook signature verification before accepting automated payment status updates.
- Run `npm run workflow:validate` and `npm run build` in CI.

## Authentication and authorization

- Credential login uses bcrypt password verification and rejects inactive users.
- Google OAuth uses NextAuth/Auth.js.
- Registration is server-side, validated and rate-limited.
- Protected pages use the Next.js 16 `proxy.ts` convention with JWT role checks.
- Every admin API route performs a fresh database role/status check.
- Customer order, invoice and document routes enforce ownership.
- Denied admin attempts are written to structured logs and `AuditLog` where the database is available.

- Credential login and registration
- OTP requests
- Document upload URL requests
- Order submission

In-memory limits are per application instance. Production requires a shared Redis-backed sliding-window implementation.

## Release gates

```bash
npm run qa:preflight
npm run env:production
npm run seo:validate
npm run prelaunch:check
```

- `qa:preflight` runs environment validation, Prisma generation, workflow validation and the production build.
- `env:production` rejects non-HTTPS/local URLs, fallback identity/contact values, placeholder providers, disabled Redis/2FA and missing legal/partner approval flags.
- `seo:validate` checks public route metadata usage, canonical generation, sitemap URLs and the robots sitemap URL against `NEXT_PUBLIC_SITE_URL`.
- `prelaunch:check` combines production environment, SEO and deployable-content checks.

Required release approval flags:

```text
LEGAL_REVIEW_APPROVED=true
PARTNER_DETAILS_VERIFIED=true
PRODUCTION_DOMAIN_APPROVED=true
DEMO_DATA_ENABLED=false
```

Set these only after the corresponding manual checks are complete.
